## 目的
- 为 fpdev 的 tests 子树统一工程结构、构建方式与依赖配置，确保跨平台、可维护、可扩展
- 指导后续新增测试/示例/临时验证工程的标准化落地

## 目录与命名规范
- 每个测试子项目至少包含：
  - `*.lpr`：入口程序（首行必须 `{$CODEPAGE UTF8}`）
  - 可选 `*.lpi`：Lazarus 工程文件
  - 可选项目级 runner：例如 `tests/fpdev.build.manager/run_tests.bat` / `tests/fpdev.build.manager/run_tests.sh`
- 不要假设每个测试子项目都自带 `buildOrTest.bat`、本地 `bin/`、本地 `lib/`
- 单元测试文件命名：命名空间 + .test.pas
- 单元测试结构：
  - 全局函数放入 TTestCase_Global
  - 类对象单独建 TTestCase_类名，测试模块公开接口（含重载）
  - 用例命名：Test_函数名 或 Test_函数名_参数1_参数2

## .lpr 约定
- 必须置顶声明：{$CODEPAGE UTF8}
- 默认使用 {$mode objfpc}{$H+}，并遵循现有代码风格
- 局部变量命名以 L 开头（团队约定）

## .lpi 约定（核心）
- 当前仓库常见模式是把维护中的测试工程输出到仓库根 `bin/` 与 `lib/`
- 以 `tests/fpdev.build.manager/test_build_manager.lpi` 为例：
  - `Target/Filename=../../bin/test_build_manager`
  - `OtherUnitFiles=../../src`
  - `UnitOutputDirectory=../../lib/$(TargetCPU)-$(TargetOS)`
- BuildModes：
  - Debug：
    - 断言/边界/溢出/栈检查开启
    - DebugInfo=DWARF3，UseHeapTrc=True
  - Release：
    - 优化 O3（OptimizationLevel=3）
    - SmartLinkUnit=True；LinkSmart=True
    - 去调试信息（等价 -CX -Xs -g-）
- RequiredPackages：
  - 所有测试工程追加 FCL
  - 仅“单元测试工程”追加 fpcunittestrunner（检测 uses fpcunit/fpcunittestrunner/TestFramework）

## 构建与脚本
- lazbuild 在 PATH 中可直接使用
- 顶层 Pascal 回归基线：
  - `bash scripts/run_all_tests.sh`
- 聚焦单个顶层测试：
  - `bash scripts/run_single_test.sh tests/test_config_management.lpr`
- 子项目级 runner 仅在该目录显式提供时使用：
  - Windows：`tests/fpdev.build.manager/run_tests.bat`
  - Linux/macOS：`bash tests/fpdev.build.manager/run_tests.sh`
- `buildOrTest.bat` 只应视为少数遗留子树的局部工具，不是新增测试工程的统一前提
- 批量构建（Release）：tests/build_all_release.bat

## 最佳实践
1) 构建工具：lazbuild
2) 输出目录：优先复用仓库根 `bin/` 和 `lib/`，并与 `.lpi` / runner 保持一致
3) 测试模式：Debug 开启内存泄漏检查（HeapTrc）、各类检查；Release 开性能优化
4) 中文输出：所有测试/示例 .lpr 顶部添加 {$CODEPAGE UTF8}
5) 全局配置：仅使用 src/项目名.settings.inc 作为宏配置

## 扩展与对齐
- 新增测试工程前，先检查目标子树是否已有现成 runner / `.lpi` 模式，再决定是否复用仓库根 `bin/` / `lib/`
- 如需使用局部 `bin/` / `lib/`，必须在对应 `.lpi` 与 runner 中显式声明，不能把它当作全仓默认
- 持续将临时验证移入 plays/，示例移入 examples/，与 tests 保持分离
